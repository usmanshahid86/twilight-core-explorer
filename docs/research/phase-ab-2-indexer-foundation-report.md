# Twilight Core Explorer Phase A/B-2 Indexer Foundation Report

## 1. Summary

Phase A/B-2 adds the first durable indexing foundation for Twilight Core Explorer.

Implemented:

- Prisma/PostgreSQL baseline.
- Generic explorer schema foundation.
- `packages/db` workspace package.
- `apps/indexer` workspace package.
- Height-range indexer CLI skeleton.
- `ingestHeight()` service using the `ChainClient` interface.
- Mandatory block, block-results, and tx-by-height source calls.
- Idempotent upserts for blocks, transactions, messages, events, accounts, and cursor.
- Single-writer Postgres advisory lock helper.
- Hash mismatch halt behavior.
- Fixture/unit tests for empty blocks, tx blocks, idempotency, cursor safety, hash mismatch, advisory lock, and route guards.

This phase intentionally does not implement CoreSlot or rewards semantic projections.

## 3. Prisma Schema Added

The schema uses PostgreSQL and Prisma Client.

Generic foundation models:

- `Block`
- `ExplorerTransaction`
- `Message`
- `Event`
- `Account`
- `IndexerCursor`
- `DecodeFailure`

Important schema choices:

- `ExplorerTransaction` avoids possible naming confusion with transaction concepts in ORM/runtime code.
- `Event.eventKey` is a deterministic unique key generated by the indexer, so null `txHash` block events remain safely idempotent.
- Raw JSON is stored on block, transaction, message, event, account, and decode-failure records where applicable.
- No CoreSlot semantic projection tables are included yet.
- No rewards semantic projection tables are included yet.
- No staking, governance, mint module, or distribution models are included.

## 4. Indexer App Skeleton

`apps/indexer` provides a minimal CLI entrypoint:

```sh
START_HEIGHT=1 END_HEIGHT=10 npm --prefix apps/indexer start
```

Required runtime env:

- `DATABASE_URL`
- `CHAIN_ID`
- `COMET_RPC_URL`
- `REST_URL`

Optional runtime env:

- `START_HEIGHT`
- `END_HEIGHT`
- `REQUEST_TIMEOUT_MS`

If `START_HEIGHT` is omitted, the CLI starts from `IndexerCursor.lastIndexedHeight + 1`.
If `END_HEIGHT` is omitted, it uses `ChainClient.getStatus().latestBlockHeight`.

The CLI is range-based only. It is not an infinite daemon yet.

## 5. Ingestion Behavior

`ingestHeight()` depends on `packages/chain-client` `ChainClient`, not direct routes.

For every height, it calls:

- `client.getBlock(height)`
- `client.getBlockResults(height)`
- `client.getTxsByHeight(height)`

Behavior:

- Stores `Block` even when there are zero transactions.
- Stores begin/end block events from `/block_results`.
- Stores transaction rows by hash.
- Stores message rows by `txHash,msgIndex`.
- Stores event rows by deterministic `eventKey`.
- Stores conservative discovered accounts.
- Stores raw source payload JSON.
- Tolerates unknown messages and events.
- Updates cursor only after all height writes succeed.

## 6. Cursor and Advisory Lock Behavior

Cursor helpers:

- `getOrCreateCursor(chainId)`
- `updateCursorSuccess(chainId, height, hash, latestChainHeight?)`
- `haltCursorHashMismatch(chainId, height, expectedHash, actualHash)`
- `haltCursorError(chainId, height, error)`

Allowed cursor statuses:

- `idle`
- `running`
- `halted_hash_mismatch`
- `halted_error`

The advisory lock helper uses a fixed two-int Postgres advisory lock:

- namespace: `847001`
- key: `20260624`

The helper uses:

- `pg_try_advisory_lock`
- `pg_advisory_unlock`

If the lock is unavailable, the indexer fails clearly before processing heights.

## 7. Idempotency Behavior

Idempotent writes use:

- `Block.height`
- `ExplorerTransaction.hash`
- `Message.txHash,msgIndex`
- `Event.eventKey`
- `Account.address`
- `IndexerCursor.chainId`

Re-ingesting the same fixture height does not duplicate block, transaction, message, or event rows.

Account `txCount` is currently conservative: first discovery creates `txCount = 1`, and re-ingestion does not increment it. A later account-activity aggregation can compute exact counts from indexed tx/message/event data.

## 8. Hash Mismatch Behavior

Before writing a fetched block, `ingestHeight()` checks whether a block already exists at the same height.

If the existing block has a hash and the fetched block has a different hash:

- cursor status becomes `halted_hash_mismatch`
- the existing block is not overwritten
- ingestion throws `HashMismatchError`
- processing stops

If a hash is missing from one source, the current implementation uses the best available block hash and does not halt solely because one side is missing.

## 10. Known Limitations

- Tests use fixtures and a mocked Prisma surface; they do not require a live Postgres test database.
- A real Prisma/Postgres integration test should be added once local Docker Compose or test DB conventions are introduced.
- The CLI is range-based and does not run as an infinite daemon.
- Account extraction uses a conservative `twilight1` prefix check with a TODO for stricter bech32 validation.
- Account `txCount` is intentionally conservative to avoid over-counting on idempotent replays.
- Message decoding is generic. Unknown message types are stored but not semantically decoded.
- Event projection is generic. CoreSlot and rewards event semantics are deferred.
- No balance snapshots are written yet.
- No generated gRPC client exists yet.
